# 엔진 Time Cap 정책 논의 화두

작성일: 2026-06-11  
상태: 논의 전 화두 정리. 아직 결정 문서가 아니다.

## 2026-06-11 1차 반영

앱 기본값은 우선 다음처럼 조정했다.

- B16: 기본 `1000ms`, 선택지 `500/1000/1500/2000/2500ms`
- B32: 기본 `2000ms`, 선택지 `1000/2000/3000/4000/5000ms`
- B64: 기본 `3000ms`, 선택지 `3000/4500/6000/7500/9000ms`

benchmark 평균값은 자동 적용하지 않고, 메뉴의 `추천[...]` 보조 지표로만 표시한다. 실제 time cap은 사용자가 선택한 값이 적용된다.

## 배경

이전 `빠른 초급`은 코드상 `16 visits / 250ms`로 제한되어 있었다.

- 위치: `shared/src/commonMain/kotlin/com/worksoc/goaicoach/shared/PlayLevel.kt`
- 과거 값: `FastBeginner.visits=16`, `FastBeginner.timeMillis=250`
- 폰 실측 예시: `request=16`, `root=2`, `elapsedMs=397`, `fill=SHORT`
- 같은 폰 benchmark 예시: B16 평균 약 `2656ms`, root avg 약 `15.0`, `SHORT`

즉 과거 기본 대국은 쾌적하지만 visits를 안정적으로 채우지 못할 수 있었다.

## 논의할 질문

1. 플레이 레벨의 time cap은 “쾌적한 응답 시간”을 우선할 것인가, “요청 visits 충족”을 우선할 것인가?
2. benchmark 평균 시간이 2초라면 실제 대국도 2초에 가깝게 올릴 것인가, 1초처럼 의도적으로 낮춘 값을 쓸 것인가?
3. `fill=SHORT`가 반복되어도 빠른 초급의 학습/대국 품질로 허용 가능한가?
4. 빠른 초급은 속도 우선, 초급 이상은 visits 충족 우선처럼 그룹별 철학을 다르게 둘 것인가?
5. time cap을 기기 benchmark 결과로 자동 보정할 것인가, 사용자가 직접 “속도 우선/균형/정확도 우선”을 고르게 할 것인가?
6. startup benchmark의 5000ms 결과를 그대로 대국 time cap에 반영해도 되는가, 별도 fast-call micro benchmark가 필요한가?
7. AI 착수와 사람 차례 리뷰/Top Moves의 time cap을 같게 둘 것인가, 사람 차례 리뷰는 더 낮은 cap으로 분리할 것인가?
8. 대국 중 `fill=SHORT` 빈도를 누적 기록해 이후 time cap을 자동 조정할 것인가?

## 정책 후보

### 후보 A: 속도 우선 고정 cap

- 빠른 초급은 `250ms` 또는 `400ms`처럼 짧은 cap을 유지한다.
- 장점: 대국 리듬이 좋고 느린 폰에서도 체감이 좋다.
- 단점: visits 미충족으로 AI 품질과 후보 평가 신뢰도가 낮아질 수 있다.

### 후보 B: benchmark 평균 기반 cap

- B16 평균이 2초면 B16 계열 cap을 2초 근처로 올린다.
- 장점: 요청 visits 충족 가능성이 높아진다.
- 단점: 초급 대국도 느려져 사용자 체감이 나빠질 수 있다.

### 후보 C: benchmark 평균의 일부만 반영

- benchmark 평균이 2초면 실제 대국 cap은 1초처럼 낮춘다.
- 장점: 품질과 속도의 절충이 가능하다.
- 단점: 어떤 비율이 적절한지 실측과 사용자 테스트가 필요하다.

### 후보 D: 사용자 선택형

- 같은 레벨이라도 `속도 우선`, `균형`, `정확도 우선`을 제공한다.
- 장점: 기기 성능과 사용자 선호를 모두 수용한다.
- 단점: UI/설정 복잡도가 증가한다.
