# B16 vs B32 시간 cap 1000ms 추가 실험

실행일: 2026-06-10

## 목적

폰에서 `빠른 초급 3단계(B16 best)`가 `초급 7단계(B32 best)`를 자주 이기는 체감이 있어, `maxTime` 부족으로 B32가 충분히 32 visits를 채우지 못하는지 확인한다.

## 조건

- command: `python3 scripts/run-katago-level-match.py --black fast_beginner:3 --white beginner:7 --black-time-ms 1000 --white-time-ms 1000 --games 50 --swap-colors --seed 20260611 --out docs/engine-match-logs/b16-vs-b32-time1000-20260610.jsonl`
- visits: B16 `16`, B32 `32`
- time cap: B16/B32 모두 `1000ms`
- deterministic: `false`
- searchThreads: `4`
- final evaluator: `400 visits / 2000ms`
- raw log: `docs/engine-match-logs/b16-vs-b32-time1000-20260610.jsonl`
- 소요 시간: 약 6분 26초

## 해석

- 맥북에서는 기존 조건에서도 B16/B32가 거의 항상 요청 visits를 채웠다.
- time cap을 1000ms로 늘려도 평균 root visits는 거의 변하지 않았다. 즉 맥북 실험에서는 `maxTime`이 아니라 `maxVisits`가 먼저 search를 끝낸다.
- 이번 1000ms 조건에서 B32 우세가 26승 24패로 약해졌지만, root visits는 동일하므로 시간 증가 때문에 강도가 약해졌다고 보기는 어렵다. 저방문수, `nnRandomize=true`, search thread, 9x9 swing에 따른 표본 흔들림으로 보는 것이 타당하다.
- 폰에서는 CPU/GPU 성능 차이로 500ms 안에 B32가 visits를 다 못 채울 가능성이 있으므로, 시간 cap 확대는 “강도 향상”보다 “느린 기기에서 visits를 안정적으로 채우는 안전장치”로 의미가 있다.

## 다음 판단

앱 기본값을 즉시 바꾸기 전에, 폰에서 AI 자동 대전 로그에 `rootVisits`, `elapsedMs`, `timeRequestMs`, `visitsRequest`를 모아 B32가 실제로 32 visits를 못 채우는지 확인하는 것이 우선이다.

만약 폰 로그에서 B32 root visits가 32 미만으로 자주 잘리면:

- B16: `16 visits / 500ms`
- B32: `32 visits / 1000ms`
- B64: `64 visits / 1500ms` 또는 `2000ms`

처럼 time cap을 기기 안정성 기준으로 올리는 방향을 검토한다.
